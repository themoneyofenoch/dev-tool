description: Deploy anything — auto-detects platform, auto-commits, auto-retries, hands-off

---

Deploy the current project. Zero decisions needed. Auto-detects platform, commits dirty state, fixes failing checks, and ships it.

`$ARGUMENTS` can be:
- An environment: `dev`, `staging`, `prod` (default: auto-detect from branch)
- A platform override: `vercel`, `netlify`, `zeabur`, `eas`, `fly`, `cloudflare`, `docker`, `capacitor`
- A flag: `--skip-checks`, `--skip-build`, `--force`
- `rollback` — rollback the last deploy

## Step 0: Load Saved Config

If `.deploy.json` exists in the project root, load it:
```json
{
  "platform": "vercel",
  "environment": "production",
  "buildDir": "dist",
  "projectId": "...",
  "lastDeployUrl": "...",
  "lastDeployCommit": "..."
}
```

Use saved values as defaults. Any `$ARGUMENTS` override takes precedence.

If `.deploy.json` does NOT exist, proceed with auto-detection. After the first successful deploy, **create `.deploy.json`** with the detected values so the next deploy is instant.

## Step 1: Environment Resolution

Determine the target environment:
1. If `$ARGUMENTS` contains `dev` → **development**
2. If `$ARGUMENTS` contains `staging` → **staging**
3. If `$ARGUMENTS` contains `prod` or `production` → **production**
4. If `.deploy.json` has `environment` → use that
5. Auto-detect from git branch:
   - `main` or `master` → **production**
   - `staging` or `release/*` → **staging**
   - anything else → **development** (preview)

## Step 2: Auto-Detect Project Type & Platform

If `.deploy.json` has a `platform`, skip detection and use it.

If `$ARGUMENTS` explicitly names a platform, use that.

Otherwise, scan the project root for these signals (in order of priority):

| Signal Found | Project Type | Deploy Target |
|---|---|---|
| `eas.json` | Expo/React Native mobile | EAS Build & Submit |
| `capacitor.config.*` | Capacitor hybrid mobile | Capacitor + native build |
| `fastlane/Fastfile` | Native mobile | Fastlane |
| `android/` + `build.gradle` | Android native | Gradle + Play Store |
| `*.xcodeproj` or `*.xcworkspace` | iOS native | xcodebuild + App Store |
| `tauri.conf.json` | Tauri desktop | Tauri CLI |
| `electron-builder.*` or `electron.vite.config.*` | Electron desktop | electron-builder |
| `Dockerfile` + no frontend signals | Backend/API service | Docker |
| `fly.toml` | Backend/API | Fly.io |
| `railway.json` or `railway.toml` | Backend/API | Railway |
| `vercel.json` or `.vercel/` or `next.config.*` | Web app/site | Vercel |
| `zeabur.json` or `.zeabur/` | Web app/site | Zeabur |
| `netlify.toml` or `.netlify/` | Web app/site | Netlify |
| `wrangler.toml` or `wrangler.json` | Workers/Pages | Cloudflare |
| `firebase.json` | Web app/site | Firebase Hosting |
| `amplify.yml` or `.amplify/` | Web app/Backend | AWS Amplify |
| `render.yaml` | Web app/Backend | Render |
| `package.json` (generic) | Web app/site | Vercel (default) |

If nothing is detected, ask: "I couldn't detect a deployment target. What platform are you using?"

## Step 3: Auto-Commit Dirty State

Check `git status --porcelain`:
- **Clean** → proceed, nothing to do.
- **Dirty** → auto-commit everything:
  1. `git add -A`
  2. Generate a commit message from the diff summary (list changed files, max 5, then "+N more")
  3. `git commit -m "<auto-generated message>"`
  4. If deploying to production, also `git push` so the commit is backed up
  5. Print: `Auto-committed: "<message>"`

Do NOT ask for permission. Just do it. The user said deploy.

Exception: if a file looks like it contains secrets (`.env`, `credentials.json`, `*.pem`, `*.key`), **do NOT commit it**. Add it to `.gitignore` instead and warn the user.

## Step 4: Pre-flight Checks

Skip if `--skip-checks` is in `$ARGUMENTS`.

1. **CLI check**: Verify the required CLI is installed for the detected platform. Install if missing:
   - Vercel: `npm install -g vercel`
   - Netlify: `npm install -g netlify-cli`
   - Zeabur: `npm install -g zeabur`
   - Cloudflare: `npm install -g wrangler`
   - Firebase: `npm install -g firebase-tools`
   - Fly.io: `curl -L https://fly.io/install.sh | sh`
   - EAS: `npm install -g eas-cli`
   - Docker: check `docker` is available
   - Railway: `npm install -g @railway/cli`
2. **Auth check**: Verify the user is logged in to the platform. If not, guide them.
3. **Security scan**: Quick check for accidentally committed secrets:
   ```bash
   git diff --staged | grep -iE '(api_key|secret|password|token|private_key).*=.*["'"'"']' || echo "clean"
   ```

## Step 5: Pre-deploy Validation (with Smart Retry)

Skip if `--skip-checks` is in `$ARGUMENTS`.

Run in parallel:
1. **Lint** — from AGENTS.md, or `npm run lint`, or equivalent
2. **Typecheck** — from AGENTS.md, or `npm run typecheck`, or `tsc --noEmit`
3. **Test** — from AGENTS.md, or `npm test`
4. **Build** — skip if `--skip-build` is in `$ARGUMENTS`. From AGENTS.md, or `npm run build`

### Smart Auto-Retry (up to 3 attempts)

If any check fails:
1. **Attempt 1**: Show the error, auto-fix it (lint errors, type errors, etc.), re-run the failed check
2. **Attempt 2**: If still failing, try a different fix approach. Re-run.
3. **Attempt 3**: If still failing, **stop**. Show a clean summary of what failed and why. Ask the user how to proceed.

If all checks pass on any attempt, proceed immediately. Don't waste the user's time.

Print: `Checks passed (attempt N/3)` — only if it took more than 1 attempt.

## Step 6: Deploy

Deploy based on the detected platform and target environment:

### Vercel
```bash
# Production
vercel --prod --yes
# Staging
vercel --yes --env=staging
# Development/Preview
vercel --yes
```

### Zeabur
```bash
# Login if needed
zeabur auth login
# Deploy (auto-detects project type)
zeabur deploy
# Deploy to specific environment
zeabur deploy --environment <dev|staging|production>
```

### Netlify
```bash
# Production
netlify deploy --prod --dir=<build-dir>
# Staging/Preview
netlify deploy --dir=<build-dir>
```

### Cloudflare Pages / Workers
```bash
# Pages (production)
wrangler pages deploy <build-dir> --project-name=<name>
# Workers (production)
wrangler deploy
# Preview
wrangler pages deploy <build-dir> --branch=<branch>
```

### Firebase Hosting
```bash
# Production
firebase deploy --only hosting
# Staging (use a Firebase target)
firebase deploy --only hosting:staging
```

### Fly.io
```bash
# Production
fly deploy
# Staging (if separate app)
fly deploy --app <staging-app-name>
```

### Railway
```bash
# Production
railway up
# Staging
railway up --environment staging
```

### Docker
```bash
# Build and push
docker build -t <image-name>:<tag> .
docker push <image-name>:<tag>
# Tag with version if git tag exists
```

### EAS (Expo/React Native)
```bash
# iOS
eas build --platform ios --profile <environment>
eas submit --platform ios --profile <environment>
# Android
eas build --platform android --profile <environment>
eas submit --platform android --profile <environment>
# All platforms
eas build --platform all --profile <environment>
eas submit --platform all --profile <environment>
```

### Capacitor
```bash
# Build web assets first
npm run build
npx cap sync
# iOS
npx cap open ios  # then Xcode build, or:
npx cap run ios --configuration <Release|Debug>
# Android
npx cap open android  # then Android Studio build, or:
npx cap run android --configuration <Release|Debug>
```

### Fastlane
```bash
# iOS
fastlane ios deploy_<environment>
# Android
fastlane android deploy_<environment>
```

### Electron
```bash
# Build and publish
npm run build  # or electron-builder --publish always
```

### Tauri
```bash
# Build release
cargo tauri build
# Or with updater
cargo tauri build --bundles updater
```

If the platform CLI is missing or the project isn't linked/configured yet, guide the user through setup before deploying.

## Step 7: Post-deploy

1. **Health check** (web apps/APIs only):
   - Curl the deployed URL and check for HTTP 200
   - If health endpoint exists (`/health`, `/api/health`, `/status`), hit that
   - Timeout after 60 seconds with retry every 10 seconds

2. **Save config**: Write `.deploy.json` with platform, environment, buildDir, deployed URL, and commit hash. Next deploy uses this.

3. **Print summary**:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Deployed!
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Platform:      <Vercel | Netlify | EAS | ...>
  Environment:   <production | staging | development>
  URL:           <url if applicable>
  Branch:        <branch>
  Commit:        <short-hash>
  Version:       <from package.json or git tag>
  Status:        ✅ live
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

4. **Rollback info**: Print how to rollback for the detected platform:
   - Vercel: `/deploy rollback` or `vercel rollback`
   - Zeabur: `zeabur deploy rollback` or rollback via Zeabur dashboard
   - Netlify: `netlify deploy --prod --alias=<previous-deploy-id>`
   - Fly.io: `fly releases` then `fly deploy --image <previous-image>`
   - EAS: `eas submit --platform <ios|android> --id <previous-build-id>`
   - Docker: re-deploy previous image tag
   - General: `git revert HEAD && /deploy`

If the health check fails, immediately print the rollback command and ask the user if they want to rollback.

## Rollback

If `$ARGUMENTS` is `rollback`:
1. Load `.deploy.json` to get platform and last deploy info
2. Execute the platform-specific rollback:
   - Vercel: `vercel rollback`
   - Zeabur: rollback via Zeabur dashboard or redeploy previous commit
   - Netlify: `netlify deploy --prod --alias=<previous-deploy-id>`
   - Fly.io: `fly deploy --image <previous-image>`
   - Docker: re-deploy previous tag
   - General: `git revert HEAD` then redeploy
3. Health check the rolled-back deployment
4. Print: `Rolled back to <commit> — <url>`
